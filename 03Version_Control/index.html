Version Control with Git

Git is a distributed version control system used for tracking changes in source code during software development. It allows multiple developers to collaborate on a project while maintaining a complete history of changes, enabling versioning, branching, and reverting to earlier states of the code.

Git Workflows

Git workflows are strategies or frameworks used by development teams to collaborate effectively using Git. Common workflows include:

  • Feature Branch Workflow: Developers work on feature-specific branches before merging them into the main branch.
  • Gitflow Workflow: A more structured approach with dedicated branches for features, releases, and hotfixes.
  • Forking Workflow: Often used in open-source projects where contributors fork the repository, make changes, and create pull requests.

Resolving Merge Conflicts

Merge conflicts occur when changes in two branches cannot be automatically merged by Git. This typically happens when multiple contributors edit the same lines in a file. Resolving conflicts involves:

  • Identifying the conflict in the affected file(s).
  • Editing the file to retain the desired changes.
  • Marking the conflict as resolved with the git add command.
  • Completing the merge with git commit.

Git Tags and Versioning

Git tags are references that point to specific commits, often used for marking release points (e.g., v1.0.0). Tags are of two types:

  • Lightweight Tags: Simple pointers to a commit.
  • Annotated Tags: Contain additional metadata like a message, timestamp, and signature.

Tags are commonly used in versioning to indicate stable releases and allow developers to reference specific versions of the codebase.
